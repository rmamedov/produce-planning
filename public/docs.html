Production Plan Priority API

API for storing and querying production priority data per filial. Shows current stock coverage, recommended production quantities, and urgency levels per product (lager) for today and tomorrow.

Base URL http://165.245.253.31

What's new in V2

All V2 changes are backward compatible — new fields are optional or nullable.

✨

1. New optional lagerfullname on each item (product full name).
2. sales_qty and produced_qty are now optional and may be null.
3. New department_id at the POST body root + as a GET filter.
4. GET responses now include lagerfullname, department_id and department_name.

🔗

The upsert key is unchanged: filial_id + history_date + lager_id. department_id is stored per item but is not part of the key (exactly one department per lager).

Data Model

Data is stored flat in the database with a unique constraint on filial_id + history_date + lager_id. Repeated writes for the same combination will update the existing record (upsert).

POST Write Priority Data

POST /api/production-plan-priority

Writes production priority data for a filial. Accepts nested structure with dates and items. Performs upsert — if a record with the same filial_id + history_date + lager_id already exists, it will be updated.

GET Read & Filter Data

GET /api/production-plan-priority?filial_id={id}

Returns production priority data for a given filial, grouped by date. Items are sorted by priority ASC, covered_hours ASC. Supports optional filters via query parameters.

Query Parameters

ParameterTypeRequiredDescription
filial_id integer required Delivery filial identifier
history_date string optional Filter by date (YYYY-MM-DD)
priority integer optional Filter by priority level (1-4)
lager_id integer optional Filter by product (lager) ID
department_id integer optional v2 Filter by department identifier

Example Requests

curl examples 
# All data for filial
curl "http://165.245.253.31/api/production-plan-priority?filial_id=3322"

# Filter by date
curl "http://165.245.253.31/api/production-plan-priority?filial_id=3322&history_date=2026-06-15"

# Filter by priority (critical only)
curl "http://165.245.253.31/api/production-plan-priority?filial_id=3322&priority=1"

# Filter by specific lager
curl "http://165.245.253.31/api/production-plan-priority?filial_id=3322&lager_id=123456"

# Filter by department (v2)
curl "http://165.245.253.31/api/production-plan-priority?filial_id=3322&department_id=17"

# Combine filters
curl "http://165.245.253.31/api/production-plan-priority?filial_id=3322&history_date=2026-06-15&priority=1"

Priority Levels

The priority field indicates the urgency level for production based on stock coverage.

ValueLevelConditionAction
1 Critical Stock coverage ≤ 1 hour Immediate production required
2 Warning Stock coverage ≤ 4 hours Plan production soon
3 Moderate Stock coverage > 4 hours Monitor, produce if capacity allows
4 Covered Fully covered till day end No action needed
💡

The covered_hours value is fractional. For example, 2.40 means stock covers 2 full hours + 40% of the 3rd hour's demand.

json
{

  "message": "No production plan data found for filial_id 9999"
}

500 Internal Server Error

Unexpected server error.

json
{
  "message": "Internal server error"
}

Notes

🔄

Data refreshes every hour, aligned with DAMS stock snapshots.

📅

snapshot_hour for tomorrow will be null until stock data becomes available.

⚠️

Items with priority = 4 are fully covered — no action needed. recommended_to_produce = 0 when current_stock_qty ≥ demand_till_day_end.

🔁

POST performs upsert: if a record with the same filial_id + history_date + lager_id already exists, all fields are updated. No duplicates are created.
